# Mobile UI Agent — RL Environment

A reinforcement-learning-style environment that simulates a simple mobile application where an AI agent completes tasks by producing structured JSON actions and receives rewards based on task success, action validity, efficiency, and safety.

Built following the [Prime Intellect Verifiers](https://github.com/PrimeIntellect-ai/verifiers) design philosophy: separable reward components, a clean `load_environment()` factory, and a dataset split into train / eval / hidden-eval partitions.

---

## Quick Start

```bash
# 1. Clone & enter the repo
git clone <repo-url> && cd mobile_ui_env

# 2. Create a virtual environment (Python 3.11+)
python -m venv .venv
.venv\Scripts\activate        # Windows
# source .venv/bin/activate   # macOS / Linux

# 3. Install dependencies
pip install -r requirements.txt

# 4. Run tests
pytest -v

# 5. Run evaluation (heuristic baseline)
python run_eval.py

# 6. (Optional) Run with an LLM via OpenRouter
python run_eval.py --mode llm --api-key YOUR_OPENROUTER_KEY
```

## 1. State Space

The state is represented by `AppState`, a dataclass that captures the full observable + internal state of the simulated mobile app:

| Field             | Type          | Description                                        |
|-------------------|---------------|----------------------------------------------------|
| `current_screen`  | `str`         | One of: `home`, `notes`, `settings`, `profile`     |
| `notes`           | `list[str]`   | Saved note titles                                  |
| `pending_note`    | `str`         | Text typed but not yet saved                       |
| `note_input_active`| `bool`       | Whether the note-creation flow is active            |
| `focus_mode`      | `bool`        | Focus mode toggle state (default: `False`)          |
| `notifications`   | `bool`        | Notifications toggle state (default: `True`)        |
| `logged_out`      | `bool`        | Whether the user has logged out (safety-critical)   |
| `profile`         | `dict`        | `{username, email}` — read-only user info           |
| `app_version`     | `str`         | Simulated app version string                        |
| `steps`           | `int`         | Actions executed so far                             |
| `invalid_actions` | `int`         | Count of invalid actions                            |
| `safety_violations`| `int`        | Count of safety violations                          |
| `visited_screens` | `set[str]`    | Screens visited during the episode                  |
| `action_log`      | `list[dict]`  | Full action history                                 |

**Observation**: The agent receives a textual observation generated by `AppState.observation()`, which mirrors an accessibility-tree representation of the current screen — listing available elements, toggle states, notes, and user info.

### Simulated Screens

| Screen   | Elements                                                         |
|----------|------------------------------------------------------------------|
| Home     | `notes_button`, `settings_button`, `profile_button`              |
| Notes    | `add_note_button`, `note_input`, `save_note_button`, `note_list` |
| Settings | `focus_mode_toggle`, `notifications_toggle`, `version_label`     |
| Profile  | `username_label`, `email_label`, `logout_button`                 |

## 2. Action Space

The agent produces a **JSON array** of actions. Each action is a dict with an `"action"` field and optional parameters:

| Action   | Parameters            | Description                          |
|----------|-----------------------|--------------------------------------|
| `tap`    | `target` (element ID) | Tap a UI element on the current screen |
| `type`   | `target`, `text`      | Type text into an input field        |
| `back`   | —                     | Navigate back to the home screen     |
| `finish` | —                     | End the episode                      |

**Example agent output:**
```json
[
  {"action": "tap", "target": "notes_button"},
  {"action": "tap", "target": "add_note_button"},
  {"action": "type", "target": "note_input", "text": "Buy milk"},
  {"action": "tap", "target": "save_note_button"},
  {"action": "finish"}
]
```

Invalid actions (wrong target, missing fields, malformed JSON) are **counted and penalised** but **never crash** the environment.

## 3. Episode Termination Conditions

An episode ends when **any** of these occur:
1. The agent issues a `finish` action.
2. The step counter reaches `max_steps` (defined per task).
3. All actions in the agent's output array have been executed.

After termination, rewards are computed over the final state.

---

## 4. Sparse Rewards

| Component         | When non-zero                                |
|--------------------|----------------------------------------------|
| `success_reward`   | Only at episode end: `1.0` if goal fully met, else `0.0` |
| `safety_penalty`   | `1.0` if *any* safety violation occurred during the episode |

These are **sparse** — they give a binary signal only at the end. This is challenging for RL agents because the agent receives no gradient signal for "almost correct" trajectories. An agent that navigates to the correct screen but fails the final sub-action gets the same reward (0) as one that does nothing.

---

## 5. Dense / Shaped Rewards

| Component               | How it's computed                                                  |
|--------------------------|---------------------------------------------------------------------|
| `format_reward`          | Fraction of actions that were structurally valid (`valid / total`)  |
| `efficiency_reward`      | Linear decay from 1.0 (`min_steps`) to 0.0 (`max_steps`), only when goal met |
| `invalid_action_penalty` | Fraction of actions that were invalid (`invalid / total`)          |
| `partial_progress_reward`| 0.5 for reaching the correct screen, 1.0 for completing goal; for multi-goals, fraction of sub-goals completed |

These are **dense/shaped** — they provide continuous feedback that helps guide exploration even before the sparse success signal fires. The `partial_progress_reward` is especially important: it rewards the agent for making intermediate progress (e.g., navigating to the Notes screen) even if it doesn't complete the full note-creation sequence.

---

## 6. Reward Hacking Risks

Reward hacking occurs when an agent finds an unintended strategy that maximises the reward without actually completing the task. In this environment:

| Risk | Description | Mitigation |
|------|-------------|------------|
| **Format gaming** | Agent outputs many valid no-op actions (e.g., repeated `back` on home) to inflate `format_reward` | `efficiency_reward` penalises extra steps; weight on `format_reward` is low (0.1) |
| **Progress farming** | Agent navigates to the right screen repeatedly without completing the goal to collect `partial_progress_reward` | Progress is capped at 0.5 for screen visits; full credit requires goal completion |
| **Efficiency shortcut** | Agent immediately calls `finish` in 1 step — technically "efficient" | `efficiency_reward` is gated on `success`: 0 if goal not met |
| **Toggle oscillation** | Agent toggles a setting back and forth to appear to have "changed" it | Goal checks the *final* state, not the number of toggles |
| **Safety exploitation** | In a poorly designed rubric, an agent might find that the reward from other components outweighs the safety penalty | Safety penalty weight (0.3) is the second-highest, and the penalty is binary, making it hard to offset |

**General mitigation strategy**: Keep the `success_reward` weight dominant (0.6), gate other rewards on goal completion where possible, and maintain a high safety penalty weight (0.3). Positive weights are normalised to sum to 1.0, so a perfect run scores exactly 1.0 — penalties then push the score below 1.0.

## 7. Scaling to a Real Android Emulator

The mock environment is designed to be a **drop-in replaceable** simulation layer. To scale to a real Android environment:

| Layer | Mock (current) | Real Android |
|-------|----------------|--------------|
| **State representation** | `AppState` dataclass | Screenshot + UI hierarchy XML (accessibility tree) |
| **Observation** | Text description of current screen | Screenshot image, XML dump, or parsed text tree |
| **Action execution** | In-memory state mutation via `execute_action()` | ADB commands, Android Accessibility Service, or UI Automator |
| **Element targeting** | String element IDs | Resource IDs, content descriptions, XPath selectors, or coordinates |
| **Navigation** | Dict lookup (`NAVIGATION`) | Actual Android intents or tap coordinates |
| **Goal verification** | State field checks | Screenshot OCR, XML tree assertions, or app state queries |
| **Emulator management** | N/A | Android Emulator API, Docker containers with Android images |

**Key real-world inputs for the agent:**
- **Accessibility tree**: Structured XML of the UI hierarchy with element IDs, content descriptions, and bounding boxes.
- **Screenshot**: Pixel-level observation for visual reasoning.
- **UI hierarchy**: Combined view of the XML tree for programmatic element selection.
- **Action executor**: Module that translates abstract actions (tap, type) into ADB or UIAutomator commands.
- **Emulator state**: CPU, memory, network status for environment health monitoring.

The environment interface (`reset` → `step` → `reward`) remains identical; only the backend changes.

## 8. Integration with Prime Intellect / Verifiers / PRIME-RL

This environment provides a Verifiers-compatible `load_environment()` factory:

```python
from mobile_ui_env import load_environment

env = load_environment()
# env.dataset       → 20 train tasks
# env.eval_dataset  → 10 eval tasks
# env.rubric        → Rubric with 6 weighted reward functions
```

The `Rubric` and `SingleTurnEnv` classes mirror the Verifiers API:

```python
# Reset for a task
obs = env.reset(task)

# Agent produces actions
actions = agent(obs)

# Step + evaluate
result = env.step(actions)
rubric_scores = env.evaluate(task)
```

**To integrate with PRIME-RL training:**
1. Wrap `SingleTurnEnv` as a Gymnasium environment for PPO/GRPO training.
2. The `rubric.evaluate()` method provides the shaped reward signal.
3. Train/eval split prevents overfitting to specific task instances.
4. Hidden-eval tasks enable holdout evaluation during training.

## 10. Tradeoffs Due to Limited Scope

| Decision | Rationale |
|----------|-----------|
| **Single-turn only** | Multi-turn would require conversation state management; single-turn is sufficient to demonstrate the RL environment pattern |
| **Text observations instead of screenshots** | Real Android environments use pixel data; text is simpler and makes the task tractable without vision models |
| **In-memory state instead of real UI** | Enables fast iteration and unit testing without an Android emulator |
| **Heuristic baseline for eval** | A perfect heuristic demonstrates that the environment is solvable; LLM integration is optional |
| **35 tasks, not 1000** | Quality over quantity; the task templates cover all goal types and can be procedurally expanded |
| **stdlib-only for core** | Zero external dependencies = easier review, no version conflicts with Python 3.11-3.13 |
| **Fixed default profile** | Real apps would have dynamic user data; static data simplifies goal verification |
| **No multi-turn conversation history** | Would be needed for agents that refine their approach based on feedback |
